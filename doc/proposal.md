# Twitch 自動掛台與忠誠點數領取 Docker 容器需求文件

## 1. 專案目標

建立一個可透過 Docker 執行的自動化服務，讓使用者設定多個 Twitch 實況主頻道後，系統能在指定實況主開台時自動啟動掛台流程，並在 Twitch 頁面出現忠誠點數獎勵按鈕時自動領取。

本專案以「瀏覽器自動化」作為主要實作方式，透過容器內的無頭或可視化瀏覽器模擬使用者已登入 Twitch 後的觀看行為與點擊操作。

## 2. 使用情境

1. 使用者準備好 Twitch 登入後的 Cookie 或瀏覽器 Storage 資料。
2. 使用者在設定檔中填入多個想要掛台的 Twitch 實況主頻道名稱。
3. 使用者透過 Dockerfile 建置映像檔，或透過 docker-compose 啟動服務。
4. 系統定期檢查設定清單中的實況主是否正在開台。
5. 當目標實況主開台時，系統自動開啟該 Twitch 頻道頁面並維持觀看狀態。
6. 當頁面出現可領取忠誠點數的按鈕時，系統自動點擊領取。
7. 系統記錄掛台狀態、領取紀錄與錯誤資訊，方便使用者追蹤。

## 3. 功能需求

### 3.1 Twitch 頻道監控

- 系統必須支援多個 Twitch 實況主頻道清單。
- 使用者必須能透過設定檔或環境變數指定頻道清單。
- 系統必須定期檢查每個指定頻道是否正在直播。
- 檢查間隔必須可設定，避免過於頻繁請求 Twitch。
- 若頻道未開台，系統應保持待命並在下一輪檢查時重新確認。
- 若頻道開台，系統應啟動或維持該頻道的掛台流程。

### 3.2 自動掛台

- 系統必須使用瀏覽器自動化工具開啟 Twitch 頻道頁面。
- 系統必須使用匯入的 Twitch Cookie 或 Storage 資料維持登入狀態。
- 系統必須盡量保持觀看頁面有效，例如：
  - 頁面成功載入直播播放器。
  - 播放器未停留在錯誤頁面。
  - 必要時重新整理異常頁面。
- 系統應支援多個頻道同時開台時的處理策略。
- 第一版預設策略為：對清單中所有已開台頻道各自開啟頁面掛台，但最大同時掛台數必須可設定。
- 若同時開台數量超過最大同時掛台數，系統應依設定檔中的頻道順序作為優先順序。

### 3.3 自動領取忠誠點數

- 系統必須在已開啟的 Twitch 頻道頁面中偵測忠誠點數獎勵按鈕。
- 當獎勵按鈕可點擊時，系統必須自動點擊領取。
- 系統必須記錄每次成功點擊領取的時間、頻道與結果。
- 若找不到按鈕，系統不得視為錯誤，應繼續監聽頁面狀態。
- 若點擊失敗，系統應記錄錯誤並在後續輪詢中再次嘗試。

### 3.4 登入狀態與 Cookie 管理

- 系統必須採用「手動匯入 Cookie 或瀏覽器 Storage」方式取得 Twitch 登入狀態。
- 系統不得要求使用者將 Twitch 密碼寫入 Docker image。
- 系統不得在日誌中輸出完整 Cookie、OAuth token、session token 或其他敏感資料。
- Cookie 或 Storage 檔案必須透過 Docker volume 掛載進容器。
- 系統必須在啟動時檢查登入資料是否存在。
- 若登入資料不存在或失效，系統應輸出清楚錯誤訊息，提示使用者重新匯入登入資料。

### 3.5 設定管理

系統應支援設定檔，建議格式為 YAML 或 JSON。設定內容至少包含：

```yaml
channels:
  - streamer_one
  - streamer_two
  - streamer_three

check_interval_seconds: 60
max_concurrent_streams: 2
headless: true
browser_data_path: /data/browser-state
log_level: info
```

設定需求：

- `channels`：必填，Twitch 頻道名稱清單。
- `check_interval_seconds`：選填，頻道開台檢查間隔，預設 60 秒。
- `max_concurrent_streams`：選填，最大同時掛台數，預設 1 或 2，實際值由實作決定。
- `headless`：選填，是否使用無頭瀏覽器，預設 true。
- `browser_data_path`：必填或具預設值，用於讀取掛載的登入狀態資料。
- `log_level`：選填，日誌層級，例如 debug、info、warn、error。

### 3.6 Docker 執行需求

專案必須提供：

- `Dockerfile`
- `docker-compose.yml`
- 範例設定檔，例如 `config.example.yml`
- 說明如何掛載 Cookie 或瀏覽器 Storage 的文件

Docker 容器需求：

- 容器內必須包含瀏覽器自動化所需的瀏覽器與依賴套件。
- 容器必須能讀取外部掛載的設定檔。
- 容器必須能讀取外部掛載的 Twitch 登入狀態資料。
- 容器應將執行日誌輸出到 stdout/stderr，方便 Docker logs 檢視。
- 若需要保存瀏覽器狀態或快取，應使用 Docker volume，而不是寫入映像檔。

範例 docker-compose 使用方向：

```yaml
services:
  twitch-watchdog:
    build: .
    container_name: twitch-watchdog
    restart: unless-stopped
    volumes:
      - ./config.yml:/app/config.yml:ro
      - ./data/browser-state:/data/browser-state
    environment:
      - CONFIG_PATH=/app/config.yml
```

## 4. 非功能需求

### 4.1 安全性

- 不得將 Twitch 帳號密碼硬編碼於程式碼、Dockerfile 或設定範例中。
- 不得在日誌中輸出敏感登入資訊。
- Cookie 或 Storage 檔案應由使用者自行保管，並建議設定適當檔案權限。
- 設定文件必須提醒使用者：匯出的登入狀態等同於帳號登入憑證，外洩可能造成帳號風險。

### 4.2 穩定性

- 系統應能長時間在容器中執行。
- 瀏覽器或頁面崩潰時，系統應嘗試重新啟動對應頁面或瀏覽器。
- 頻道離線時，系統應關閉或釋放該頻道頁面資源。
- 系統應避免無限制開啟新頁面造成記憶體耗盡。

### 4.3 可觀測性

- 系統必須輸出主要狀態日誌：
  - 服務啟動
  - 設定載入成功或失敗
  - 登入狀態檢查結果
  - 頻道開台或離線狀態變更
  - 開始掛台
  - 停止掛台
  - 忠誠點數領取成功或失敗
  - 瀏覽器錯誤或重啟
- 日誌應避免過度冗長，並可透過 `log_level` 調整。

### 4.4 合規與限制

- 本工具應只用於使用者自己擁有或授權使用的 Twitch 帳號。
- 文件必須提醒使用者自行確認 Twitch 服務條款與社群規範。
- 系統不應包含繞過驗證、隱藏自動化行為、規避平台偵測或大量帳號操作等功能。
- 系統不應支援批量登入多帳號或大量擴張觀看數的用途。

## 5. 建議技術方向

本需求文件不強制指定語言，但基於瀏覽器自動化與 Docker 部署便利性，建議候選方案如下：

- Node.js + Playwright
- Python + Playwright

建議優先考慮 Playwright，原因：

- 支援 Chromium 瀏覽器自動化。
- 支援保存與載入 browser storage state。
- 容器化部署文件與基礎映像成熟。
- 適合偵測頁面元素並執行點擊操作。

## 6. 初版範圍

初版應包含：

1. 讀取設定檔中的多個頻道清單。
2. 讀取手動匯入的 Twitch 登入狀態。
3. 定期檢查頻道是否開台。
4. 針對開台頻道開啟瀏覽器頁面掛台。
5. 自動偵測並點擊忠誠點數領取按鈕。
6. 支援最大同時掛台數與頻道優先序。
7. 提供 Dockerfile 與 docker-compose.yml。
8. 提供基礎日誌。

初版不包含：

- 自動輸入 Twitch 帳號密碼登入。
- 多帳號管理。
- 繞過驗證或反偵測機制。
- Web UI 管理介面。
- 通知系統，例如 Discord、Telegram、Email。

## 7. 後續可擴充功能

- 加入 Discord 或 Telegram 通知。
- 加入簡易 Web UI 顯示目前掛台狀態。
- 支援更細緻的頻道優先序與排程規則。
- 支援健康檢查端點，例如 `/healthz`。
- 支援匯出領點統計資料。
- 支援容器內瀏覽器狀態診斷工具。

## 8. 驗收標準

完成後應符合以下條件：

1. 使用者可使用 Dockerfile 成功建置映像檔。
2. 使用者可使用 docker-compose 啟動服務。
3. 使用者可透過設定檔指定至少兩個 Twitch 頻道。
4. 系統可讀取掛載的 Twitch 登入狀態資料。
5. 當指定頻道開台時，系統會自動開啟該頻道頁面掛台。
6. 當忠誠點數按鈕出現時，系統會自動點擊領取。
7. 領取成功、失敗與頻道狀態變化會被記錄於日誌。
8. 容器重新啟動後，系統可再次讀取既有設定與登入狀態並恢復運作。

## 9. 待確認事項

以下事項需在實作前或實作初期進一步確認：

1. Twitch 頻道開台狀態要透過頁面檢查、Twitch 官方 API，或第三方公開資訊來源判斷。
2. Cookie 或 Storage 的匯出格式與匯入流程。
3. 忠誠點數按鈕在不同語系介面下的定位方式。
4. 是否需要支援非 headless 模式以便除錯。
5. 預設最大同時掛台數應設定為多少。
6. 是否需要提供測試用 mock 頁面或整合測試策略。
